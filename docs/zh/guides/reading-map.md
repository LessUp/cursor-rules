# 项目导读

`cursor-rules` 的核心价值，不是“提供一堆能复制的提示词”，而是把 AI 协作约束沉淀成**可版本控制、可审查、可持续维护**的规则资产。阅读这套站点时，最有效的方法不是顺着目录逐页浏览，而是先建立心智模型，再回到规则目录做筛选。

## 建议阅读顺序

| 阶段 | 去哪里 | 你会得到什么 |
| --- | --- | --- |
| 1. 定位 | 首页 / 本页 | 为什么根目录 `.mdc` 是产品本体，以及站点的公共叙事 |
| 2. 学院 | [规则哲学](../academy/rule-philosophy) | 为什么规则库比 prompt 集合更适合团队长期协作 |
| 3. 架构 | [系统架构总览](../architecture/system-overview) | 从规则文件到 `rules.json`、到 Pages 展示的完整数据链路 |
| 4. 研究 | [相关工作](../research/related-work) | 这个项目在同类开源实践中的位置、差异与优势 |
| 5. 采用 | 回到首页目录 | 根据语言、前端、后端或工程主题挑选规则组合 |

## 三类读者的最短路径

### 面试官 / 架构评审

先看首页，再看[系统架构总览](../architecture/system-overview)和[相关工作](../research/related-work)。这两页会直接回答“它解决什么问题”“它与同类实践相比先进在哪里”“它的工程设计是否自洽”。

### 准备接入的团队

先看[规则哲学](../academy/rule-philosophy)，再回到首页浏览目录。这样能先统一采用原则，再决定最小规则集合。

### 维护者与贡献者

在读完公开文档后，再跳到 `docs/openspec/`。公开站点负责解释产品定位，OpenSpec 负责项目控制、治理边界与维护工作流。

## 站点不是 README 的镜像

README 负责最短公开入口；站点负责建立信任、解释架构与降低采用摩擦；OpenSpec 负责项目控制。三者分工明确，才不会互相稀释信息密度。
