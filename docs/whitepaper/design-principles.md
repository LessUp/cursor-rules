# 设计原则

本文档阐述 Cursor Rules 的核心设计哲学和原则。

## 不是 Prompts 杂货铺

Cursor Rules 与其他规则仓库的本质区别：

| 特性 | Cursor Rules | 普通 Prompts 仓库 |
|------|--------------|-------------------|
| 版本控制 | ✅ Git 追踪 | ❌ 复制粘贴 |
| 结构化格式 | ✅ YAML frontmatter | ❌ 纯文本 |
| 适用范围 | ✅ globs 定义 | ❌ 模糊适用 |
| 自动化校验 | ✅ 构建时校验 | ❌ 无校验 |
| 可组合性 | ✅ 分层设计 | ❌ 孤立片段 |

## 规则设计哲学

### 原则 1：规则应该是约束，而非指令

好的规则定义**边界**，而非**步骤**。

```markdown
# ❌ 不好的写法
每次写函数时，先写注释，再写参数，再写返回值...

# ✅ 好的写法
函数必须有类型注解和文档字符串。
```

### 原则 2：规则应该可验证

规则的内容应该能够通过代码审查或自动化工具验证。

```markdown
# ❌ 不可验证
代码应该优雅

# ✅ 可验证
函数不超过 50 行，嵌套不超过 3 层
```

### 原则 3：规则应该有明确范围

使用 `globs` 定义规则的适用范围，避免过度泛化。

```yaml
---
description: React 组件最佳实践
globs: **/*.tsx, **/*.jsx, components/**/*
---
```

### 原则 4：规则应该可组合

不同规则之间不应有硬依赖，可以独立使用或组合使用。

```
python.mdc + fastapi.mdc ✅  组合使用
react.mdc + vue.mdc     ❌  通常不会同时使用
```

## 分层设计详解

### 语言层

提供语言级别的基础规范：
- 类型系统使用
- 命名约定
- 代码风格
- 常见模式

示例：`typescript.mdc`、`python.mdc`、`java.mdc`

### 框架层

在语言基础上叠加框架特定约定：
- 项目结构
- 组件模式
- 框架 API 使用
- 最佳实践

示例：`react.mdc`、`spring.mdc`、`fastapi.mdc`

### 工程层

跨领域的工程实践：
- 容器化
- 数据库设计
- 测试策略
- CI/CD

示例：`docker.mdc`、`database.mdc`

## 规则冲突处理

当多个规则对同一场景有不同建议时：

1. **框架规则优先**：框架特定约定覆盖语言通用规范
2. **显式优于隐式**：项目根目录的 `.cursorrules` 文件可覆盖所有规则
3. **用户决策**：最终由开发者根据具体情况判断

## 维护原则

1. **增量更新**：新增规则时保持向后兼容
2. **定期审查**：移除过时或不常用的规则
3. **文档同步**：规则变更时同步更新相关文档
4. **测试覆盖**：校验脚本应覆盖所有规则格式
