# 设计决策

本文记录 `cursor-rules` 架构中的关键设计决策，解释每项选择的动机、考量和取舍。这是理解本项目工程理念的最佳切入点。

---

## ADR-001：规则文件位于仓库根目录

**决策**：所有 `.mdc` 文件位于仓库根目录，不放入 `rules/` 子目录。

**背景**：初期版本曾将规则放在 `rules/` 子目录。用户引用规则时需要完整路径，不够简洁。

**考量**：

| 选项 | 优点 | 缺点 |
|------|------|------|
| 根目录（当前方案） | 路径最短，GitHub 仓库首页直接展示 | 根目录文件较多 |
| `rules/` 子目录 | 目录结构更"整洁" | 使用时路径更长 |

**决定理由**：`.mdc` 文件是本库的**核心产品**，让它们在根目录"第一眼可见"是有意为之的设计——访问 GitHub 仓库时，用户直接看到所有可用规则，降低发现成本。

> 这个决策一旦确定，不应轻易变动。**改变文件位置会破坏所有已有的引用链接**。

---

## ADR-002：`rules.json` 是生成产物，不可手动编辑

**决策**：`docs/public/assets/rules.json` 由 `npm run build:catalog` 生成，禁止手动编辑。

**背景**：规则目录需要以结构化格式供前端 Vue 组件消费。

**考量**：如果允许手动编辑，`rules.json` 可能与 `.mdc` 源文件产生分歧（源文件更新了，JSON 忘记同步）。这类"隐性不一致"是维护噩梦的来源。

**决定理由**：单一数据源原则（Single Source of Truth）。`.mdc` 文件是数据的权威来源，`rules.json` 是其派生产物。所有对规则的修改必须通过修改 `.mdc` 文件 + 重新运行 `build:catalog` 完成。

**约束**：`.gitignore` 中**不**忽略 `rules.json`——它是构建产物但需要提交，因为 GitHub Pages 的静态托管无法执行 CI 生成步骤（设计为无需 CI 也能运行站点）。

## ADR-003：不引入前端框架（无 React、Vue 应用层）

**决策**：文档展示层使用 VitePress + Vue 组件，不引入独立的 React/Next.js 等前端框架。

**背景**：有人建议将规则目录页面迁移到 React，以支持更复杂的交互（搜索、过滤、收藏）。

**考量**：

- 引入框架意味着引入构建工具链、包依赖、版本升级负担
- VitePress 内置的 Vue 组件能力已足够实现所需交互
- 静态站点的维护成本远低于 SPA

**决定理由**：`cursor-rules` 定位为**归档级别稳定的规则库**，而非持续迭代的产品。引入框架会带来持续的维护成本，与"长期稳定"的定位冲突。VitePress 是满足需求的最简方案。

## ADR-004：规则验证在构建时而非运行时执行

**决策**：规则验证（frontmatter 完整性、字段合规性）在 CI 构建阶段执行，不在客户端 JavaScript 中执行。

**背景**：早期考虑在前端渲染规则卡片时进行数据验证，以便为不合规规则显示警告标记。

**考量**：构建时验证的优势在于失败快、反馈即时。`npm test` 失败意味着不合规规则永远无法到达用户。

**决定理由**：把问题拦截在源头（开发时）比在展示层处理更可靠。这也简化了前端逻辑——前端可以假设 `rules.json` 中的数据总是完整合规的。

## ADR-005：分类系统采用扁平枚举

**决策**：规则分类使用扁平的预定义枚举（7 个类别），不支持自定义标签（tags）。

**背景**：用户曾建议增加 `tags` 字段，支持更灵活的多维度标注（如 `[testing, linting, ci]`）。

**考量**：

- Tags 灵活但会导致分类体系分散（`test` vs `testing` vs `unit-test` 这类问题无法避免）
- 扁平枚举约束分类数量，强迫内容作者做决策，而不是无限堆 tag
- 7 个类别已覆盖主流技术栈场景

**决定理由**：简单可维护性优于灵活性。扁平枚举可以在 `categories.mjs` 中集中维护，前后端共享单一数据源，无需同步。

> 如果未来需要更细粒度的分类，应该扩展枚举而不是引入 tags 系统。

## ADR-006：文档站点与工程脚本共存于同一仓库

**决策**：`docs/` 站点、`scripts/` 工程脚本、`.mdc` 规则文件都在同一个 Git 仓库中。

**背景**：有人建议将文档站点分离到独立仓库（`cursor-rules-docs`），使规则库保持"纯净"。

**考量**：单仓库方案保持了规则与其展示逻辑的同步性。修改规则后可以在同一个 PR 中同时更新文档。

**决定理由**：`cursor-rules` 的核心价值在于"规则可以被直接浏览和发现"，文档站点是核心用户体验的一部分，不应该与规则源分离。
