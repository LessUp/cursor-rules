# Cursor Rules 终局收敛式重构设计

## 1. 背景与问题定义

当前仓库的真实形态并不是“架构腐化的大型应用”，而是一个以 `.mdc` 规则文件为核心产物的规则库仓库，外围附带一个校验脚本、README、GitHub Pages 站点，以及少量 AI 协作配置。现状的主要问题不在于业务代码复杂，而在于**产品叙事、控制文档、站点呈现、AI 协作资产、版本锚定与自动化边界没有被收敛成一个统一的终局形态**。

这会带来几类长期漂移风险：

- **文档与站点漂移**：README、GitHub Pages、规则元信息之间存在重复表达，后续容易不同步。
- **控制文档缺失**：缺少一组稳定、低冗余、能服务于“收尾与归档”的项目控制文档。
- **AI 配置碎片化**：仓库已有 `CLAUDE.local.md`、`.claude/settings.local.json`、`.claude/skills/verify`，但没有统一的项目级 AI 协作入口。
- **工程化边界未收紧**：CI/CD 当前不算复杂，但仍缺少基于真实变更面的触发约束，也没有把站点构建与规则目录之间的关系产品化。
- **仓库定位不够锋利**：它既像规则合集，又像文档站点，又像实验仓库，缺少“归档级稳定产品仓库”的明确边界。

## 2. 目标与非目标

### 2.1 目标

本次重构的目标是把仓库收敛为一个**可归档级稳定的 Cursor / AI 规则库产品仓库**：

1. **规则资产层稳定**：根目录 `.mdc` 文件保持为唯一核心产物，命名、元信息、校验与分类方式统一。
2. **用户入口清晰**：README 和 GitHub Pages 分工明确，前者负责仓库入口与快速采用，后者负责规则浏览、筛选、对比与复制。
3. **控制文档成体系**：以 OpenSpec 风格建立少量但高价值的项目控制文档，替代散乱说明。
4. **AI 协作可交接**：为 Claude、Copilot 及后续 GLM 接手提供低歧义、低漂移的仓库级指令与约束。
5. **工程化轻量但严格**：只保留能真正降低漂移和回归风险的自动化，不引入表演性复杂度。

### 2.2 非目标

本次重构**不做**以下事情：

- 不把仓库改造成多包工程、Monorepo 或伪分层应用。
- 不引入重型前端构建系统，只为静态站点增加复杂度。
- 不为了“看起来企业化”而扩展多套 CI、发布、版本流水线。
- 不引入需要长期维护的 MCP 服务，除非存在明确的外部系统集成需求。
- 不改变根目录 `.mdc` 文件作为用户直接复制使用入口的既有契约。

## 3. 目标状态与边界定义

仓库在重构后的目标边界分为四层：

1. **规则资产层**：`*.mdc` 与验证脚本，负责规则本身的正确性、稳定性与可维护性。
2. **产品呈现层**：README 与 GitHub Pages，负责“发现、筛选、理解、采用”规则，不再承担内部控制文档的职责。
3. **项目控制层**：OpenSpec 风格文档，定义仓库定位、架构边界、开发流程、版本锚定、文档治理与收尾标准。
4. **AI 协作层**：`AGENTS.md`、`CLAUDE.md`、Copilot 指令、项目内 Skills 与必要的 Claude 配置，负责把仓库知识显式化。

这四层共同构成“终局收敛式重构”的完整边界。所有实现都必须围绕这四层服务，而不是扩展出新的复杂子系统。

## 4. 目标目录与信息架构

### 4.1 仓库目录原则

- 根目录继续保留全部 `.mdc` 文件，保证现有用户复制/引用路径不失效。
- `docs/` 只保留面向 GitHub Pages 与规范文档的高价值内容，不堆积 changelog、碎片记录与重复说明。
- 项目控制文档集中到 `docs/openspec/`。
- 设计与计划文档集中到 `docs/superpowers/specs/`，服务于本轮重构与后续接手。

### 4.2 目标目录草案

```text
/
├── *.mdc
├── README.md
├── AGENTS.md
├── CLAUDE.md
├── package.json
├── scripts/
│   ├── validate-rules.mjs
│   └── build-rule-catalog.mjs
├── docs/
│   ├── index.html
│   ├── 404.html
│   ├── assets/
│   │   └── rules.json
│   ├── openspec/
│   │   ├── project-positioning.md
│   │   ├── architecture.md
│   │   ├── workflow.md
│   │   └── ai-tooling.md
│   └── superpowers/
│       └── specs/
└── .github/
    └── workflows/
        ├── validate.yml
        └── pages.yml
```

其中 `build-rule-catalog.mjs` 和 `docs/assets/rules.json` 是本次设计的关键：**GitHub Pages 不再手写维护全部规则卡片，而是消费由规则文件生成的目录数据**，从而减少手工同步与内容漂移。

## 5. Phase-by-Phase 设计

### 5.1 Phase 1：激进审查与规范化重构

这一阶段的重点不是“改造应用架构”，而是**清理叙事层和控制层的腐化**。

#### 目录与架构

- 保留根目录 `.mdc` 的扁平结构，因为这是本项目最直接、最稳定、最易采用的分发方式。
- 不引入 `src/`、`packages/` 之类与仓库真实形态不匹配的层级。
- 通过生成式目录数据代替手工站点内容，解决 Pages 与规则文件之间的耦合问题。

#### 文档治理

- README 重写为对外入口文档：价值主张、选规则方式、快速开始、校验方式、站点入口。
- 新增 `docs/openspec/` 控制文档，承载内部规范，避免 README 继续膨胀。
- 删除或避免新增低价值 changelog、过程笔记、重复性 FAQ。
- 对现有 `.mdc` 做系统审查，修复描述、标题、范围、语言风格与低信噪比表述。

#### Bug 与一致性

- 统一 `.mdc` 的 frontmatter 与标题风格。
- 审查 `globs`、描述文本、规则段落是否与文件定位一致。
- 为站点与 README 使用同一份规则目录数据源，避免“改了规则却忘记更新展示层”的结构性 bug。

#### 依赖与版本锚定

- 统一 Node 版本策略，让 `package.json`、CI、文档使用同一个支持窗口。
- 所有自动化脚本继续保持零或极低依赖，避免为静态站点引入构建链。

### 5.2 Phase 2：工程化与 GitHub 深度集成

这一阶段的目标是把 GitHub 层面的集成压缩到“够用且稳”的程度。

#### Workflow 精简

- 保留两个工作流：`validate.yml` 与 `pages.yml`。
- 为工作流添加真实路径过滤，只在 `.mdc`、脚本、站点或工作流本身变化时触发。
- 避免无意义矩阵、跨版本并行、缓存复杂化与多余状态徽章。

#### GitHub 仓库元信息

- 使用 `gh` 更新 description、homepage 与 topics。
- 让 GitHub 仓库主页与 GitHub Pages 链接指向一致的产品定位。
- 如果仓库 about 文案与 README 首屏叙事不一致，以仓库定位文档为准统一修正。

#### 开发流程固化

- 在 `docs/openspec/workflow.md` 中定义轻量工作流：单分支主线、小批量提交、以 `/review` 为核心的审查节奏、尽量减少长期分支。
- 明确何时需要 `/review`、何时只需本地验证、何时允许直接落主线。
- 强调“面向收尾”的开发方式：优先消除漂移和歧义，而不是扩展功能表面积。

### 5.3 Phase 3：AI 工具链与 Vibe Coding 配置

这一阶段的原则是：**优先使用高信噪比、低维护成本的仓库级指令；谨慎引入额外协议与插件。**

#### Copilot / Claude 指令

- 运行 `/init` 生成并定制 `.github/copilot-instructions.md`，让 Copilot 知道仓库的真实边界和编辑原则。
- 新增 `AGENTS.md`，作为跨模型、跨工具的仓库级协作总入口。
- 新增 `CLAUDE.md`，整合当前 `CLAUDE.local.md` 中真正属于仓库而非个人偏好的内容。
- 重新判断 `CLAUDE.local.md` 是否仍应留在仓库；若只表达个人习惯，则应删除或转移出共享配置面。

#### Claude 配置与项目内 Skills

- 保留并文档化 `.claude/settings.local.json` 中针对 `.mdc` 的自动验证钩子，解释为什么虽然名为 `local` 但被项目使用。
- 保留现有 `verify` Skill，因为它与仓库核心产物高度契合。
- 仅在确有价值时新增一个轻量 Skill，例如围绕规则编写/修订的审查清单；避免堆积技能目录。

#### MCP 与 CLI Skills 的取舍

- 本仓库当前没有持续访问外部系统、数据库或 SaaS API 的硬需求，因此默认**不引入 MCP**。
- 能通过仓库内脚本、静态文档、CLI Skill 解决的问题，一律优先 CLI Skill。
- 如果后续需要接 GitHub 项目板、外部知识库或规则发布系统，再单独设计 MCP，而不是本轮预埋。

#### LSP / 编辑环境

- 只增加与当前栈强相关的最小配置，例如针对 JS 脚本的 `jsconfig` 或针对编辑器的一致性设置。
- 不为了完成清单而硬配无效 LSP；对于 `.mdc` 这类非主流格式，应优先依赖结构校验器和规范文档，而不是假装存在成熟语言服务器。

#### Copilot Plugin

- 仅在确认某个 Copilot Plugin 能显著提升 `.mdc` 编写、校验或站点维护效率时才引入。
- 若没有明显收益，则明确记录“不配置插件是刻意选择”，避免未来重复评估。

### 5.4 Phase 4：收尾计划与 YOLO 执行

该阶段不是再设计，而是把前 3 个阶段固化为一份**可直接执行、可交接给 GLM 的结构化 Todo 列表**，并按依赖顺序推进。

## 6. GitHub Pages 设计

GitHub Pages 不应继续扮演 README 的镜像页，而应成为本仓库的**规则目录与采用入口**。

### 6.1 页面职责

- 首页展示仓库定位、规则总量、分类、使用方式和精选规则。
- 提供按语言/框架/工程主题筛选规则的浏览能力。
- 展示每条规则的描述、适用 glob、分类与复制入口。
- 明确引导用户返回仓库 README、Issues、贡献入口。

### 6.2 数据来源

- 站点规则数据来自 `build-rule-catalog.mjs` 对 `.mdc` frontmatter 与正文标题的解析结果。
- 站点不再维护单独的手工规则清单。
- 生成产物写入 `docs/assets/rules.json`，由静态页面直接消费。

### 6.3 设计方向

- 视觉上保留当前简洁、渐变、轻卡片风格，但提升产品表达与信息层级。
- 结构上弱化“宣传口号”，强化“规则即产品目录”的感知。
- 文案采用中英双语或关键区块双语，匹配仓库现有对外受众。

## 7. 版本与质量策略

- 所有质量门槛围绕“规则正确、展示一致、说明清晰”三点建立。
- 统一 Node 版本锚定，避免 `package.json`、CI、文档各写一套。
- 校验优先级：`.mdc` 结构正确 > 展示层不漂移 > 文档与控制规范一致。
- 对静态站点采用“无构建或极轻构建”的策略，避免发布链膨胀。

## 8. 执行顺序与任务分解

以下任务列表服务于后续 implementation plan 与实际执行：

1. 盘点并分类所有 `.mdc` 规则，确定站点目录模型与分类维度。
2. 统一 Node 版本锚定与脚本入口，补足站点目录生成脚本。
3. 重构 GitHub Pages，使其消费生成式规则目录数据。
4. 重写 README，压缩重复内容，强化对外入口职责。
5. 建立 `docs/openspec/` 控制文档体系。
6. 重构或删除不合适的共享 AI 配置，新增 `AGENTS.md` 与 `CLAUDE.md`。
7. 生成并定制 Copilot 指令文件。
8. 评估并落实最小可行的 Claude Skill / LSP / 插件策略。
9. 精简 GitHub Actions 触发条件与版本策略。
10. 使用 `gh` 更新仓库元信息与 Pages 相关设置。
11. 全量验证规则、站点、文档与配置的一致性。
12. 产出最终 Todo 清单与对 GLM 的交接说明。

## 9. 风险与约束

- 最大风险不是实现失败，而是**做成一堆看似完整、实际增加维护面的配置集合**。
- 若某项 AI / 编辑器配置无法证明其价值，应宁可不加。
- 若某项文档不能减少歧义或降低后续接手成本，应宁可不写。
- 所有“终局状态”都必须建立在**更少文件、更少重复、更少特殊例外**之上。

## 10. 通过标准

当以下条件同时满足时，本轮重构视为达到目标：

1. 规则文件、站点与 README 之间不存在结构性漂移源。
2. 项目控制文档完整但精简，能指导后续收尾与归档。
3. 仓库级 AI 协作指令齐全，且没有明显互相冲突。
4. GitHub 工作流与仓库元信息已收敛到真实需要的最小集合。
5. 后续 GLM 接手时，不需要重新理解仓库定位即可继续推进。
